# FrameForge — Product Specification & AI Prompt Manual
**Version 1.0 — Photography Layout & Export Tool**

---

## PART I — PRODUCT OVERVIEW

### 1.1 What FrameForge Is

FrameForge is a browser-based (pure HTML + JS, no install) visual composition tool for photographers. It takes a raw photograph and a JSON settings file — generated by an external AI session — and renders a fully composed, print/web-ready graphic that can be previewed and exported as a high-quality PNG.

It is **not** a design tool you operate with a mouse. It is a **rendering engine with a preview interface**. The design decisions happen in an AI conversation. FrameForge faithfully executes those decisions.

### 1.2 Intended Workflow

```
[Photographer + AI session]
    → Concept development (mood, palette, typography, layout logic)
    → Per-image settings generation (JSON)
    → JSON file saved locally

[FrameForge App]
    → Load project JSON
    → Upload source images (any filenames)
    → Drag each image from the Image Tray onto its frame to assign
    → Preview each frame
    → Export all as high-quality PNG
```

> **Note on filenames:** The AI model may use placeholder names (`photo-001.jpg`, etc.) in the JSON. Exact filename matching is a convenience — when names don't match, the user manually assigns images using the Image Tray drag-and-drop interface (see §3.9).

### 1.3 Core Design Principles

- **JSON is the source of truth.** Every visual decision lives in the settings file.
- **The app never overrides JSON.** What the AI specified is what renders.
- **Consistency is structural.** A project has shared globals; frames override only what differs.
- **Export quality is non-negotiable.** PNG output must match pixel-perfect the canvas spec.
- **No backend, no accounts.** Everything runs in the browser. Projects persist via localStorage.

## PART II — APPLICATION ARCHITECTURE

### 2.1 Technology Stack

| Layer | Choice | Rationale |
|---|---|---|
| Runtime | Pure HTML5 + Vanilla JS (ES2022 modules) | No install, no build step, works offline |
| Rendering | HTML5 Canvas API | Pixel-accurate, exportable to PNG at any resolution |
| Font Loading | Google Fonts API (dynamic) | AI can specify any GF family by name |
| Storage | localStorage (project index + JSON blobs) | Persistent across sessions, no server |
| Export | `canvas.toBlob()` → FileSaver pattern | High-quality PNG, respects DPI multiplier |
| File I/O | FileReader API | Load images + JSON from local disk |

### 2.2 Module Structure

```
frameforge/
├── index.html                  # Single entry point
├── app.js                      # App bootstrap + router
├── modules/
│   ├── project.js              # Project state management
│   ├── renderer.js             # Canvas rendering engine (core)
│   ├── layers.js               # Layer type implementations
│   ├── export.js               # PNG export + filename logic
│   ├── fonts.js                # Dynamic Google Fonts loader
│   ├── storage.js              # localStorage CRUD
│   └── validator.js            # JSON schema validation + error reporting
├── ui/
│   ├── shell.js                # App shell (sidebar, toolbar, canvas area)
│   ├── filmstrip.js            # Multi-frame navigator + drop targets
│   ├── inspector.js            # Frame metadata panel (read-only)
│   ├── dropzone.js             # Image upload + JSON import UI
│   └── image-tray.js           # Draggable image thumbnail tray
└── styles/
    ├── base.css                # Reset + CSS custom properties
    ├── shell.css               # Layout
    └── components.css          # UI components
```

### 2.3 Application States

```
EMPTY → PROJECT_LOADED → IMAGES_MATCHED → PREVIEW_READY → EXPORT_DONE
                                 ↑                 ↓
                            (re-upload)       (re-preview)
```

- **EMPTY**: No project loaded. Shows import zone.
- **PROJECT_LOADED**: JSON parsed and validated. Frames listed. Images not yet matched.
- **IMAGES_MATCHED**: All frame images resolved. Ready to render.
- **PREVIEW_READY**: Canvas rendered for current frame. Export unlocked.
- **EXPORT_DONE**: All PNGs generated and downloaded.

## PART III — JSON SETTINGS SCHEMA

This is the complete specification of the JSON format that the AI generates and FrameForge consumes. Every field is mandatory unless marked `optional`.

### 3.1 Top-Level Structure

```json
{
  "project":     { ... },
  "export":      { ... },
  "globals":     { ... },
  "frames":      [ ... ],
  "image_index": [ ... ]
}
```

`image_index` is generated by the AI model and is `optional` for the renderer, but strongly expected. See §3.8.

---

### 3.2 `project` Object

```json
"project": {
  "id": "amazon-series-01",
  "title": "El Último Gran Paraíso",
  "version": "1.0",
  "created": "2026-03-25",
  "author": "optional — string"
}
```

| Field | Type | Description |
|---|---|---|
| `id` | string | Slug-style identifier. Used in export filenames. |
| `title` | string | Human-readable project name. Shown in app UI. |
| `version` | string | Schema version. Must be `"1.0"` for this spec. |
| `created` | string | ISO date. Informational only. |
| `author` | string | `optional` |

---

### 3.3 `export` Object

```json
"export": {
  "target": "instagram-portrait",
  "width_px": 1080,
  "height_px": 1350,
  "dpi": 300,
  "scale_factor": 2,
  "format": "png",
  "filename_pattern": "{project_id}_{frame_index}_{frame_id}"
}
```

| Field | Type | Description |
|---|---|---|
| `target` | string | Canonical name from target library (see §3.3.1). Informational + used for UI label. |
| `width_px` | integer | Canvas width in CSS pixels. |
| `height_px` | integer | Canvas height in CSS pixels. |
| `dpi` | integer | Metadata DPI for print targets. `72` for screen, `300` for print. |
| `scale_factor` | integer | Export multiplier. `2` = 2× canvas size (retina). `3` for ultra-high-res print. |
| `format` | string | Always `"png"` in v1.0. |
| `filename_pattern` | string | Token-based filename. Tokens: `{project_id}`, `{frame_index}`, `{frame_id}`, `{target}`, `{date}`. |

#### 3.3.1 Canonical Target Names

| `target` value | Width | Height | Notes |
|---|---|---|---|
| `instagram-square` | 1080 | 1080 | 1:1 |
| `instagram-portrait` | 1080 | 1350 | 4:5 |
| `instagram-story` | 1080 | 1920 | 9:16 |
| `facebook-feed` | 1200 | 630 | Landscape |
| `facebook-cover` | 820 | 312 | Banner |
| `print-a4-portrait` | 2480 | 3508 | A4 at 300dpi |
| `print-a4-landscape` | 3508 | 2480 | A4 at 300dpi |
| `custom` | any | any | Requires explicit `width_px` + `height_px` |

---

### 3.4 `globals` Object

Globals define defaults shared across all frames. Any property in globals can be overridden at the frame level.

```json
"globals": {
  "background_color": "#000000",
  "font_defaults": {
    "family": "Playfair Display",
    "weight": 700,
    "color": "#FFFFFF",
    "opacity": 1.0
  },
  "logo": {
    "src": "logo.png",
    "position": "top-left",
    "width_pct": 8,
    "opacity": 0.9
  },
  "safe_zone_pct": 5
}
```

| Field | Type | Description |
|---|---|---|
| `background_color` | hex string | Canvas background. Visible if image doesn't fill frame. |
| `font_defaults` | object | Fallback typography for any text layer missing explicit font props. |
| `logo` | object | `optional` — Watermark/logo config applied to all frames unless overridden. |
| `safe_zone_pct` | number | `optional` — Percentage inset from edges treated as safe area. Visual guide only. |

---

### 3.5 `frames` Array

Each element in `frames` represents one image in the series.

```json
"frames": [
  {
    "id": "frame-01",
    "image_src": "amazon-river-aerial.jpg",
    "layers": [ ... ]
  }
]
```

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique identifier within project. Used in filenames. |
| `image_src` | string | Identifier for the source image. Used as a label; actual image is resolved by manual assignment (§3.9) or filename match as fallback. |
| `layers` | array | Ordered array of layer objects. Rendered bottom-to-top. |
| `logo` | object | `optional` — Frame-level logo override. Merges with globals logo. |
| `background_color` | hex string | `optional` — Frame-level background override. |

---

### 3.6 Layer System

Layers are the fundamental building block. Every visual element is a layer. Layers are rendered in array order (index 0 = bottom).

#### 3.6.1 Common Layer Fields

Every layer, regardless of type, includes:

```json
{
  "id": "layer-unique-id",
  "type": "...",
  "opacity": 1.0,
  "blend_mode": "normal",
  "visible": true,
  "locked": false
}
```

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique within frame. Used for debugging. |
| `type` | string | One of: `image`, `overlay`, `text`, `shape`, `stats_block`, `logo` |
| `opacity` | float | `0.0–1.0`. Default `1.0`. |
| `blend_mode` | string | CSS/Canvas blend mode: `normal`, `multiply`, `screen`, `overlay`, `soft-light`. Default `normal`. |
| `visible` | boolean | `optional`. Default `true`. |

---

#### 3.6.2 `image` Layer

The base photograph. Typically the first layer in every frame.

```json
{
  "id": "base-photo",
  "type": "image",
  "src": "amazon-river-aerial.jpg",
  "fit": "cover",
  "position": {
    "x_pct": 50,
    "y_pct": 50
  },
  "opacity": 1.0
}
```

| Field | Type | Description |
|---|---|---|
| `src` | string | Image identifier from JSON. Resolved via manual assignment first (§3.9), then filename match. |
| `fit` | string | `cover` (fill, crop), `contain` (fit, letterbox), `fill` (stretch). Default `cover`. |
| `position.x_pct` | float | Horizontal focal point percentage. `0` = left edge, `100` = right edge, `50` = center. |
| `position.y_pct` | float | Vertical focal point percentage. `0` = top, `100` = bottom, `50` = center. |

---

#### 3.6.3 `overlay` Layer

A color wash over the image. Used for darkening, tinting, duotone effects.

```json
{
  "id": "dark-overlay",
  "type": "overlay",
  "color": "#000000",
  "opacity": 0.45,
  "blend_mode": "normal",
  "gradient": {
    "enabled": true,
    "direction": "to-bottom",
    "from_opacity": 0.0,
    "to_opacity": 0.75,
    "from_position_pct": 40,
    "to_position_pct": 100
  }
}
```

| Field | Type | Description |
|---|---|---|
| `color` | hex string | Overlay fill color. |
| `opacity` | float | Overall opacity of the overlay layer. |
| `gradient` | object | `optional` — If present, replaces flat fill with a gradient. |
| `gradient.direction` | string | `to-bottom`, `to-top`, `to-right`, `to-left`, `to-bottom-left`, `to-bottom-right` |
| `gradient.from_opacity` | float | Opacity at gradient start. |
| `gradient.to_opacity` | float | Opacity at gradient end. |
| `gradient.from_position_pct` | float | Where gradient starts (% of canvas height/width). |
| `gradient.to_position_pct` | float | Where gradient ends (% of canvas height/width). |

---

#### 3.6.4 `text` Layer

A single text element. For multi-line text, each semantic block (headline, subtitle, body) should be its own text layer.

```json
{
  "id": "main-title",
  "type": "text",
  "content": "El último",
  "font": {
    "family": "Playfair Display",
    "style": "normal",
    "weight": 700,
    "size_pct": 9.5,
    "line_height": 1.1,
    "letter_spacing_em": -0.02,
    "color": "#FFFFFF",
    "opacity": 1.0
  },
  "position": {
    "mode": "zone",
    "zone": "bottom-left",
    "offset_x_pct": 6,
    "offset_y_pct": -28
  },
  "max_width_pct": 80,
  "align": "left",
  "shadow": {
    "enabled": true,
    "color": "#000000",
    "opacity": 0.4,
    "blur_px": 12,
    "offset_x": 0,
    "offset_y": 2
  },
  "transform": {
    "italic_override": true
  }
}
```

##### Font Fields

| Field | Type | Description |
|---|---|---|
| `family` | string | Google Fonts family name. Exact match required. |
| `style` | string | `normal` or `italic`. |
| `weight` | integer | `100`, `300`, `400`, `500`, `600`, `700`, `800`, `900`. |
| `size_pct` | float | Font size as % of canvas height. Scales with any canvas size. **Always use this, never px.** |
| `line_height` | float | Unitless line height multiplier. |
| `letter_spacing_em` | float | Letter spacing in em units. Negative for tighter tracking. |
| `color` | hex string | Text fill color. |
| `opacity` | float | Text opacity independent of layer opacity. |

##### Position Fields

| Field | Type | Description |
|---|---|---|
| `mode` | string | `zone` or `absolute` |
| `zone` | string | Used when `mode: zone`. One of 9 zones: `top-left`, `top-center`, `top-right`, `middle-left`, `middle-center`, `middle-right`, `bottom-left`, `bottom-center`, `bottom-right` |
| `offset_x_pct` | float | Horizontal nudge from zone anchor, as % of canvas width. Positive = right. |
| `offset_y_pct` | float | Vertical nudge from zone anchor, as % of canvas height. Positive = down, negative = up. |
| `x_pct` | float | Used when `mode: absolute`. X position as % of canvas width. |
| `y_pct` | float | Used when `mode: absolute`. Y position as % of canvas height. |

##### Shadow Fields (`optional`)

| Field | Type | Description |
|---|---|---|
| `enabled` | boolean | |
| `color` | hex string | Shadow color. |
| `opacity` | float | Shadow opacity. |
| `blur_px` | integer | Blur radius in pixels at 1× scale. |
| `offset_x` | integer | Horizontal shadow offset. |
| `offset_y` | integer | Vertical shadow offset. |

    "x_pct": 0,

| Field | Type | Description |

| `position.x_pct` | float | Left edge of shape as % of canvas width. |
| `position.y_pct` | float | Top edge of shape as % of canvas height. |
| `dimensions.width_pct` | float | Width as % of canvas width. |

| `dimensions.height_px` | integer | `optional` — Fixed height in px (useful for thin lines). One of `height_pct` or `height_px` required. |
| `border_radius_px` | integer | `optional` — Rounded corners for rectangles. |

A structured block of one or more stat items, each with a large number and a small label. As seen in the example images ("483 ESPECIES DE AVES · 10M+ HECTÁREAS DE SELVA · 32 ETNIAS").

  "items": [
    {
      "value": "483",
      "label": "ESPECIES DE AVES",
      "value_font": {
        "family": "Playfair Display",
        "weight": 700,
        "size_pct": 5.5,

      },
      "label_font": {
        "family": "Inter",
        "weight": 400,
        "size_pct": 1.2,

| Field | Type | Description |
|---|---|---|
| `layout` | string | `horizontal` or `vertical` |
| `gap_pct` | float | Space between items as % of canvas width. |
| `items` | array | One or more stat items. |
| `items[].value` | string | The large display number/text. |
| `items[].label` | string | The small uppercase label. |
| `items[].value_font` | object | Font config for value. Same fields as text layer font. |

### 3.8 `image_index` Array

An optional top-level array generated by the AI model. It serves as a reference table documenting how each image was used in the project and the design rationale behind each frame.

```json
"image_index": [
  {
    "key": "photo-001.jpg",
    "frame_id": "frame-01",
    "frame_index": 1,
    "description": "Hero shot. Father and son on horseback against the Andean skyline. Wide negative space above used for the main headline. Warm amber gradient anchored at the bottom protects the stats block. Focal point set to x_pct: 45, y_pct: 35."
  },
  {
    "key": "photo-002.jpg",
    "frame_id": "frame-02",
    "frame_index": 2,
    "description": "Supporting frame. Close-up of the child's face. Minimal overlay — the expression carries the frame. Headline moved to top-left to avoid covering the face."
  }
]
```

| Field | Type | Required | Description |
|---|---|---|---|
| `key` | string | Yes | Image identifier used in the JSON (may be a placeholder name). |
| `frame_id` | string | Yes | The `id` of the frame that uses this image. |
| `frame_index` | integer | Yes | 1-based position of the frame in the series. |
| `description` | string | Yes | 1–4 sentences: the image's role in the series, key design decisions made, why this layout was chosen for this specific image. |

**Display:** FrameForge shows `image_index` in the Inspector panel. The entry for the currently active frame is highlighted.

**Validation:** `image_index` is `optional` — its absence does not block rendering. If present, the validator checks that each entry has a `key` and `frame_id`.

### 3.9 Image Assignment

**Problem:** AI models cannot know the photographer's actual file names. The JSON may contain placeholder names (`photo-001.jpg`, `frame1.jpg`) that don't match the photographer's real files (`IMG_4582.jpg`, `DSC_0045.jpg`). Filename-based auto-matching is therefore unreliable.

**Solution:** Manual drag-and-drop image assignment.

#### How it works

1. The user loads any image files (names don't matter).
2. All loaded images appear in the **Image Tray** — a panel at the bottom of the filmstrip column showing thumbnail previews.
3. The user drags an image from the tray onto a **filmstrip frame** or directly onto the **canvas** to assign it to that frame.
4. The frame immediately re-renders with the assigned image.
5. Assignments are persisted to `localStorage` and restored on next session.

#### Fallback

If an uploaded image's filename exactly matches a `frame.image_src` value, it is used automatically — no drag-and-drop needed. Manual assignment takes priority over filename matching.

#### Assignment scope

Manual assignment applies to the frame's primary `image_src` slot. Logo layers (which reference real asset files by name) continue to use filename-based matching.

#### Storage key

`frameforge_assignments_{projectId}` → `{ [frameIndex]: imageKey }`

### 4.1 Layout Structure

## PART V — RENDERING ENGINE SPECIFICATION

### 5.1 Render Pipeline

For each frame, the renderer executes layers in order:

```
1. Clear canvas to background_color
2. For each layer (bottom to top):
   a. Save canvas state
   b. Apply layer opacity + blend_mode
   c. Execute layer-type renderer
   d. Restore canvas state
3. Composite final canvas
```

### 5.2 Coordinate System

All positions and sizes defined in JSON use **percentage units** relative to canvas dimensions. The renderer converts at render time:

```
px = pct / 100 * canvas_width      (for x, width)
px = pct / 100 * canvas_height     (for y, height)
font_px = size_pct / 100 * canvas_height
```

This ensures layouts scale perfectly to any canvas size, including high-DPI export.

### 5.3 Zone Anchor System

The 9 zones map to canvas anchor points:

```
top-left      → (0, 0)
top-center    → (50%, 0)
top-right     → (100%, 0)
middle-left   → (0, 50%)
middle-center → (50%, 50%)
middle-right  → (100%, 50%)
bottom-left   → (0, 100%)
bottom-center → (50%, 100%)
bottom-right  → (100%, 100%)
```

`offset_x_pct` and `offset_y_pct` are added to the anchor. For bottom zones, negative `offset_y_pct` moves upward (away from the edge), which is the natural mental model.

### 5.4 Image Rendering (`cover` mode)

```
scale = max(canvas_w / img_w, canvas_h / img_h)
draw_w = img_w * scale
draw_h = img_h * scale
offset_x = (canvas_w - draw_w) * (position.x_pct / 100)
offset_y = (canvas_h - draw_h) * (position.y_pct / 100)
ctx.drawImage(img, offset_x, offset_y, draw_w, draw_h)
```

### 5.5 Text Wrapping

Text layers with `max_width_pct` must word-wrap. The renderer:
1. Converts `max_width_pct` to pixels.
2. Iterates words, measuring each with `ctx.measureText()`.
3. Breaks lines when accumulated width exceeds max width.
4. Draws each line offset by `line_height * font_size_px`.

### 5.6 Export Resolution

At export time, the canvas is scaled by `scale_factor`:

```
export_width = width_px * scale_factor
export_height = height_px * scale_factor
```

All px-converted values during rendering are also multiplied by `scale_factor`. The result is a high-resolution PNG that, when displayed at `width_px × height_px`, is sharp at retina density.

## PART VII — VALIDATION SPECIFICATION

### 7.1 Validation Rules

The validator runs on JSON import and reports errors/warnings before any rendering.

#### Errors (block rendering)
- Missing required top-level keys (`project`, `export`, `globals`, `frames`)
- `frames` array is empty
- Frame missing `id` or `image_src`
- Layer missing `id` or `type`
- Unknown layer `type` value
- `export.width_px` or `export.height_px` is zero or negative
- Duplicate `id` values within a frame's layer array

#### Warnings (allow rendering, shown in inspector)
- `export.scale_factor` > 3 (performance warning)
- Font family not available on Google Fonts (will fall back to serif)
- `image_src` file not yet uploaded (renders with placeholder)
- `size_pct` > 20 (unusually large — likely a mistake)
- `opacity` outside 0–1 range (clamped automatically)

### 7.2 Validation Output Format

```json
{
  "valid": false,
  "errors": [
    { "path": "frames[2].layers[0]", "code": "MISSING_FIELD", "message": "Layer missing required field: type" }
  ],
  "warnings": [
    { "path": "frames[0].layers[3]", "code": "IMAGE_NOT_UPLOADED", "message": "Image 'amazon.jpg' not yet matched" }
  ]
}
```

## PART X — CONCEPT BUILDER (NEW PROJECT BRIEF)

### 10.1 Purpose

The Concept Builder is an in-app 5-step wizard that lets photographers describe their project, import images, and export a complete AI package. The package — brief text + layout mockups + thumbnail sheets — is given to an AI model along with the actual image files to generate the FrameForge JSON settings file.

**The photographer provides:** title, platform, story, tone, and image files.
**The AI decides:** color palette, typography, overlay style, layout, and all text content.

**Full workflow:**
```
[FrameForge — Concept Builder Wizard]
    → Step 1: Title, platform, story, notes
    → Step 2: Tone & style
    → Step 3: Import images (file picker or drag/drop → images also go to Image Tray)
    → Step 4: Export Package (brief .txt + 3 layout mockup PNGs + thumbnail sheets)
    → Step 5: Copy Prompt (paste into AI session)

[AI Model session]
    → Attach exported package + actual image files
    → AI reviews brief + mockups + thumbnails, proposes concept, outputs JSON

[FrameForge — Main App]
    → Load JSON → images already in tray from Step 3 → Preview → Export
```

---

### 10.2 Trigger

A **"New Brief"** button is always visible in the toolbar. Clicking it opens the Concept Builder wizard.

The wizard can also be opened from the **"≡ My Briefs"** header link inside an open ConceptBuilder session (closes the current wizard and opens BriefManager).

---

### 10.3 Wizard Steps

**Auto-save:** The wizard silently saves brief state to localStorage on every step navigation and on every keystroke in the Title field. Auto-save is skipped if the title field is empty. No toast or status indicator is shown.

#### Step 1 — Title & Story

|---|---|---|

| Field | Required | Notes |
|---|---|---|

### XI.6 Data Model

**localStorage keys:**

| Key | Value |
|---|---|
| `frameforge_briefs` | JSON array of index entries `{ id, title, slug, platform, imageCount, created, updated }` |
| `frameforge_brief_{id}` | Full brief JSON object |

**Full brief schema:**

```js
{
  id:            string,   // "brief_1711234567890" — timestamp-based
  title:         string,
  slug:          string,   // derived from title at save time
  platform:      string,   // e.g. "instagram-portrait" or "custom"
  customW:       string,
  customH:       string,
  story:         string,
  notes:         string,
  tone:          string,
  toneCustom:    string,
  thumbRatioVal: string,
  thumbCustomW:  string,
  thumbCustomH:  string,
  imageCount:    number,   // last known image count — reminder only
  created:       string,   // ISO date string "YYYY-MM-DD"
  updated:       string,   // ISO date string — updated on every auto-save
}
```

Images are **not** persisted. `imageCount` is stored as a reminder only.

### XI.7 Brief Lifecycle

1. User opens ConceptBuilder ("New Brief" or "Edit" from BriefManager)
2. Once a title is typed, the wizard auto-saves on every step navigation and every title keystroke (silent — no toast)
3. Saved briefs appear immediately in "My Briefs"
4. User can re-open any brief to edit, export, or copy the prompt
5. Duplicating creates a copy for variation without overwriting the original
6. Delete removes from localStorage entirely — no undo
